# Apps

Apps are permanent, production-grade environments that serve traffic on their own domains. Unlike sandboxes, which are temporary and accessed via path prefixes, apps own their domain and run as fully independent WordPress sites.

## Creating an app

```bash
wp rudel app create --domain=client-a.com
```

This creates an isolated environment, maps the domain to it, and sets up the database tables and `wp-content` directory. The name is derived from the domain automatically, or you can specify one:

```bash
wp rudel app create --domain=client-a.com --name="Client A Website"
```

To start with a copy of the host site's data:

```bash
wp rudel app create --domain=client-a.com --clone-all
```

The same granular clone flags available to sandboxes also work here:

```bash
wp rudel app create --domain=client-a.com --clone-db --clone-themes
```

If you want to create an app from an existing sandbox or app instead of cloning from the host, use `--clone-from`:

```bash
wp rudel app create --domain=client-b.com --clone-from=client-a-com-a1b2
```

## Sandboxes from apps

Apps are meant to be the long-lived runtime target. Sandboxes are where you make and test changes.

Create a sandbox from an app:

```bash
wp rudel app create-sandbox client-a-com-a1b2 --name="Client A Feature Work"
```

That gives you a disposable sandbox with the app's database and `wp-content`, but rewritten to sandbox URLs and prefixes.

## Backups and deploys

Apps do not use sandbox snapshots. Instead, they use app backups and deploys.

Create a backup:

```bash
wp rudel app backup client-a-com-a1b2 --name=before-redesign
```

List backups:

```bash
wp rudel app backups client-a-com-a1b2
```

Restore a backup:

```bash
wp rudel app restore client-a-com-a1b2 --backup=before-redesign --force
```

Deploy a sandbox into an app:

```bash
wp rudel app deploy client-a-com-a1b2 --from=client-a-feature-a1b2 --backup=before-launch --force
```

Deploy always creates or uses an app backup first, then replaces the app's database and `wp-content` with the sandbox state while rewriting URLs back to the app domain.

## Domain management

Each app has at least one domain. You can add more:

```bash
wp rudel app domain-add client-a-com-a1b2 --domain=www.client-a.com
```

And remove them:

```bash
wp rudel app domain-remove client-a-com-a1b2 --domain=www.client-a.com
```

An app must always have at least one domain. Domains are stored in the app's `.rudel.json` metadata and compiled into a `domains.json` lookup file that the bootstrap reads on every request. Domains are normalized to lowercase and matched case-insensitively at runtime, so `Client-A.com` and `client-a.com` are treated as the same mapping.

## How routing works

When a request comes in, the bootstrap checks the `HTTP_HOST` header against the domain map before any sandbox detection runs. If the host matches an app's domain, the environment is activated and WordPress boots in that app's context. The app's `WP_SITEURL` and `WP_HOME` are set to the domain itself, not a path prefix.

This means apps work exactly like standalone WordPress sites from the browser's perspective. No path prefix in URLs, no special cookies. Just a normal domain pointing to a fully isolated WordPress environment.

## Managing apps

List all apps:

```bash
wp rudel app list
```

View details:

```bash
wp rudel app info client-a-com-a1b2
```

Destroy an app (removes database tables, `wp-content`, and domain mappings):

```bash
wp rudel app destroy client-a-com-a1b2 --force
```

## Admin bar

When you're logged into an app's wp-admin, a blue indicator with the app ID appears in the admin bar. Unlike sandboxes, there's no exit link since apps are permanent and don't have a "host" to return to.

## Differences from sandboxes

Apps and sandboxes share the same isolation layer but differ in intent:

| | Sandboxes | Apps |
|---|---|---|
| **Lifecycle** | Temporary, disposable | Permanent |
| **Routing** | Path prefix + cookie | Domain-based |
| **Recovery** | Snapshots | Backups |
| **Promote to host** | Yes | No |
| **Deploy from sandbox** | N/A | Yes |
| **Git/GitHub** | Yes | No |
| **Email** | Blocked by default | Allowed by default |
| **Use case** | Development, testing, AI agents | Multi-tenant hosting, client sites |

Both use the same `rudel_` table prefix safety guard, the same `wp-content` isolation, and the same authentication salt isolation. The intended lifecycle is: create an app, branch off a sandbox from it when you need to work, then deploy the finished sandbox back into the app.

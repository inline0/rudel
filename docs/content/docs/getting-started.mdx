# Getting Started

Getting Rudel running is straightforward once the network shape is correct. The main requirement is a WordPress multisite network that runs on subdomains. From there, Rudel installs its runtime bootstrap, creates real sites for sandboxes and apps, and stores its operational metadata in WordPress tables.

## Prerequisites

Before installing the plugin, make sure the host WordPress instance can support the kind of sites Rudel is going to create:

- WordPress multisite is enabled
- the network uses subdomains, not subdirectories
- you can write to `wp-config.php`

Rudel expects that foundation to be in place before installation. The URLs it creates are normal multisite URLs that WordPress owns directly.

## Install Rudel

### Via Composer

```bash
composer require rudel/rudel
wp plugin activate rudel
```

### Via zip

```bash
wp plugin install rudel.zip --activate
```

## Verify the runtime install

On activation, Rudel installs an early runtime bootstrap into `wp-config.php`. That gives it one guaranteed place to wire itself in before WordPress finishes booting.

```bash
wp rudel status
```

If `status` is clean, the bootstrap is in place, the runtime tables are available, and Rudel can manage the network normally.

## Create a sandbox

Sandboxes are the disposable side of Rudel. Use them for feature work, QA, debugging, demos, migrations, and any workflow where you want to change site state freely and throw it away later if needed.

```bash
wp rudel create --name=alpha
```

That command creates three things together:

1. a real multisite site
2. a Rudel environment record
3. a generated environment directory with scoped tooling and environment-owned files

To see the browser URL WordPress assigned, ask WordPress directly:

```bash
wp site list --fields=blog_id,url
```

You can then run commands directly against that site:

```bash
wp --url=http://alpha.localhost option get siteurl
```

Or work from the generated environment directory, which gives you a scoped `wp-cli.yml` plus the environment-owned files Rudel manages directly for that site, such as bootstrap files, logs, snapshots, backups, and workspace artifacts:

```bash
cd /path/to/wp-content/rudel-environments/alpha-1234
wp option get siteurl
```

## Create an app

Apps are the stable side of the lifecycle. They are also real multisite sites, but Rudel treats them as long-lived runtime targets that can accumulate backups, domain metadata, and deployment history over time.

```bash
wp rudel app create --name=demo --domain=demo.example.test
```

The `--domain` value becomes part of the Rudel app metadata. The runtime site also shows up through normal multisite tools such as `wp site list`. WordPress owns the site itself, and Rudel owns the app lifecycle around it.

If the app should also carry a tracked GitHub code source, add it at creation time:

```bash
wp rudel app create --name=demo --domain=demo.example.test --github=inline0/demo-theme --branch=main --dir=themes/demo-theme
```

That tracked repository, branch, and directory are then inherited automatically when you create sandboxes from the app.

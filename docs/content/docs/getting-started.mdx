# Getting Started

## Installation

### Via Composer (Recommended)

```bash
composer require rudel/rudel
```

Composer installations receive updates through Composer.

Upload the zip to your WordPress plugins directory and activate the plugin, or install via WP-CLI:

```bash
wp plugin install rudel.zip --activate
```

## Setup

On activation, Rudel adds a single line to your `wp-config.php`:

```php
require_once __DIR__ . '/wp-content/plugins/rudel/bootstrap.php';
```

This bootstrap file runs before `wp-settings.php` and handles all environment detection and isolation. When no environment is active, it does nothing and WordPress boots normally with zero overhead.

Verify everything is in place:

```bash
wp rudel status
```

## Configuration

Rudel works out of the box with sensible defaults. If you need to customize the CLI command name or the URL path prefix, define these constants in `wp-config.php` before the plugin loads:

| Constant | Default | Description |
|---|---|---|
| `RUDEL_CLI_COMMAND` | `rudel` | WP-CLI command name. Set to e.g. `sandbox` for white-label use. |
| `RUDEL_PATH_PREFIX` | `__rudel` | URL path prefix for sandbox access. |
| `RUDEL_GITHUB_TOKEN` | (none) | GitHub personal access token for push/PR integration. |

## Create a sandbox

```bash
wp rudel create --name="my-sandbox"
```

This creates a disposable environment with its own database and `wp-content`. To work with it via WP-CLI, `cd` into its directory:

```bash
cd /path/to/wp-content/rudel-environments/my-sandbox-a1b2
wp post list
```

Or clone your entire host site:

```bash
wp rudel create --clone-all
```

See [Creating Environments](/docs/environments/creating) for all options including database engines and GitHub integration.

## Create an app

```bash
wp rudel app create --domain=client-a.com
```

This creates a permanent environment routed by domain. Point your DNS at the server, and `client-a.com` serves a fully independent WordPress site from the same installation.

See [Apps](/docs/environments/apps) for domain management and configuration.

## Accessing environments in the browser

Sandboxes are accessible via a path prefix that works without any DNS or server configuration:

```
example.com/__rudel/my-sandbox-a1b2/
```

When you visit a sandbox URL, Rudel automatically sets a browser cookie so that subsequent requests, including `/wp-admin/`, stay in the sandbox context. To switch back to the host, append `?adminExit` to any URL.

Apps are accessed by their domain directly. No path prefix, no cookie needed.

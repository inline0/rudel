# Introduction

Rudel is a WordPress isolation layer. It creates fully isolated environments within a single WordPress installation, each with their own database, `wp-content` directory, and WP-CLI scope. Environments can be disposable sandboxes for development and testing, or permanent apps that serve real traffic on their own domains.

## Why Rudel?

WordPress doesn't have a good answer for isolated environments. Testing changes on a live site is risky. Spinning up Docker or a local dev stack is heavy. Staging environments are expensive to maintain. And if you're running multiple client sites, you're managing multiple WordPress installations with separate databases, updates, and backups.

Rudel solves all of this with a single plugin. One WordPress install can serve any number of isolated environments. Sandboxes for development, apps for production. Each one invisible to the others, each one disposable or permanent as needed.

For AI coding agents, Rudel provides the missing piece: a safe, scoped WordPress context where agents can make changes, run commands, and iterate without risk to real data. Every environment gets a `CLAUDE.md` with the context agents need.

## Two modes

**Sandboxes** are disposable, temporary environments for development, testing, and AI agent workflows. They're accessed via a URL path prefix (`/__rudel/{id}/`) and can be created, snapshotted, promoted to host, and destroyed in seconds. They integrate with git and GitHub for branch-based workflows.

**Apps** are permanent, production-grade environments routed by domain. Point `client-a.com` at your WordPress install, create an app for that domain, and it runs as a fully independent site. No multisite overhead, no shared plugin headaches.

Both modes share the same isolation layer: isolated database tables (with a `rudel_` prefix that can never collide with the host), isolated `wp-content`, per-environment authentication salts, and object cache namespacing. Sandboxes block outbound email by default; apps allow it by default.

## How it works

On activation, Rudel adds one line to `wp-config.php` that loads a bootstrap file before WordPress boots. For every incoming request, the bootstrap checks whether it targets an environment (via domain, path prefix, cookie, header, or subdomain). If it does, it rewires WordPress's constants to point to that environment's isolated resources. If it doesn't, WordPress boots normally with zero overhead.

Each environment lives in its own directory with a `.rudel.json` metadata file, a `wp-cli.yml` that auto-scopes CLI commands, and an isolated `wp-content` with its own themes, plugins, and uploads.

## Key features

Rudel is designed around a few principles: environments should be instant to create, fully isolated, and managed entirely through WP-CLI and the PHP API.

Beyond basic creation and destruction, sandboxes support point-in-time snapshots, reusable templates, zip-based export and import, promotion to replace the host site, per-environment error logging, and git/GitHub integration for branch-based workflows with merge-based cleanup. Apps support domain mapping with multiple domains per app.

Everything is accessible through the `Rudel\Rudel` static class for programmatic use, and through the `wp rudel` CLI for command-line workflows.

## Requirements

Rudel requires PHP 8.0 or later, WordPress 6.4 or later, and write access to `wp-config.php` for the initial bootstrap installation. If you plan to use the SQLite engine, you'll also need the PDO SQLite PHP extension, which is built into most PHP installations.

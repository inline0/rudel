# Introduction

Rudel manages WordPress sandboxes and apps as real subdomain multisite sites.

That means:

- every environment has a real `blog_id`
- every environment has a real site URL
- `wp-admin`, login, REST, assets, and uploads behave like normal WordPress site requests
- Rudel adds orchestration, metadata, snapshots, backups, templates, deploys, and worktree-aware code flows on top

## Two environment types

**Sandboxes** are short-lived sites for change work, QA, debugging, and agent-driven development.

**Apps** are long-lived sites that hold stable runtime state and support backups, deploys, and rollback.

Both use the same runtime model: WordPress multisite.

## Runtime shape

Rudel keeps two things in sync:

1. WordPress multisite sites  
Each Rudel environment is backed by a real multisite site.

2. Rudel runtime records  
Rudel stores environment metadata, app identity, domain metadata, worktrees, and deployment history in WordPress tables.

Rudel does not emulate sites through synthetic browser paths. The browser contract is the real site URL.

## What Rudel adds

- environment creation and destruction
- app creation from the same runtime model
- app-derived sandboxes
- snapshots for sandboxes
- backups for apps
- deploy and rollback flows
- reusable templates
- GitHub push and PR workflows
- policy metadata such as owner, labels, protection, and expiry

## Requirements

Rudel requires:

- PHP 8.0 or later
- WordPress 6.4 or later
- a subdomain multisite network
- write access to `wp-config.php` during initial runtime installation

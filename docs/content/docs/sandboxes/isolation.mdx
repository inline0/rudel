# How Isolation Works

Each Rudel sandbox is fully isolated from the host WordPress installation and from other sandboxes. This page explains the mechanisms that make this possible.

## Database isolation

Rudel supports three database engines for sandboxes:

### MySQL (default)

Each sandbox gets its own table prefix (e.g., `wp_a1b2c3_`) within the host MySQL database. The host's tables are untouched; the sandbox only reads and writes its own prefixed tables. Benefits:

- **Full compatibility**: All WordPress plugins and queries work without translation.
- **Fast cloning**: Tables are copied natively within MySQL using `CREATE TABLE ... LIKE`.
- **No extra dependencies**: Uses the existing MySQL connection.

### SQLite (experimental)

With `--engine=sqlite`, each sandbox gets its own SQLite database file (`wordpress.db`). Benefits:

- **Portable**: The entire database is a single file that can be copied or exported.
- **No shared state**: Completely separate from the host MySQL database.
- **No credentials**: No connection strings or passwords.

The SQLite integration is handled by the [WordPress SQLite Database Integration](https://github.com/WordPress/sqlite-database-integration), which translates WordPress's MySQL queries to SQLite on the fly.

### Subsite (multisite only)

With `--engine=subsite` on a WordPress multisite installation, each sandbox is created as a native multisite sub-site via `wpmu_create_blog()`. Benefits:

- **Native multisite**: Uses WordPress's built-in per-site table structure (`wp_N_posts`, `wp_N_options`, etc.).
- **Shared users**: The multisite user table is shared across the network.
- **Clean destruction**: `wpmu_delete_blog()` handles all table cleanup.

The subsite engine requires the host to be a multisite installation. All three engines provide the same wp-content isolation: each sandbox gets its own themes, plugins, and uploads directory.

## wp-content directory

Each sandbox has its own `wp-content` directory containing:

- `themes/`: Sandbox-specific themes (copied from host)
- `plugins/`: Sandbox-specific plugins
- `uploads/`: Isolated media uploads
- `mu-plugins/`: Sandbox must-use plugins

WordPress core files (`wp-includes/`, `wp-admin/`) are shared from the host installation. Only the content layer is isolated.

## Table prefix

Each sandbox uses its own table prefix derived from the sandbox ID. For MySQL sandboxes, this is the primary isolation mechanism. For SQLite sandboxes, it provides defense-in-depth.

## Bootstrap mechanism

The `bootstrap.php` file in each sandbox sets WordPress constants before WordPress boots:

- `WP_CONTENT_DIR` and `WP_CONTENT_URL`: Point to the sandbox's wp-content.
- `WP_PLUGIN_DIR` and `WPMU_PLUGIN_DIR`: Sandbox plugin directories.
- `WP_TEMP_DIR`: Sandbox temp directory.
- `UPLOADS`: Sandbox upload path.
- `DB_DIR` and `DB_FILE`: Point to the sandbox's SQLite database (SQLite engine only).

These constants are set before `wp-settings.php` runs, so WordPress sees the sandbox as its entire world.

## open_basedir restriction

The bootstrap sets PHP's `open_basedir` to restrict file access:

```php
ini_set('open_basedir', implode(PATH_SEPARATOR, [
    $sandbox_path,          // Sandbox directory (read/write)
    $wp_core_path,          // Shared WP core (read)
    $rudel_plugin_dir,      // Rudel plugin directory (read)
    sys_get_temp_dir(),     // PHP temp directory
    '/tmp',                 // Fallback temp directory
]));
```

In CLI mode, the WP-CLI tool's own directory is also added to the allowlist.

Once set, any file operation outside these paths fails silently. The host's `wp-config.php`, other sandboxes, and the host database files are all inaccessible from within a sandbox.

## Authentication salts

Each sandbox gets its own set of authentication salts, deterministically derived from the sandbox ID at boot time. This means sessions and cookies from one sandbox are not valid in another, preventing cross-sandbox session hijacking.

## WP-CLI scoping

Each sandbox includes a `wp-cli.yml` file:

```yaml
path: /var/www/html
require:
  - /path/to/sandboxes/sandbox-{id}/bootstrap.php
```

Any `wp` command executed from within the sandbox directory automatically uses that sandbox's database and content. No flags needed.

## Security model summary

| Attack vector | Defense |
|---|---|
| Path traversal (`../../`) | `open_basedir` blocks access outside allowlist |
| Sandbox escape to host DB | MySQL: table prefix isolation; SQLite: file-based, `open_basedir` restricts access |
| Cross-sandbox access | Each sandbox's `open_basedir` only includes its own path |
| Agent modifies its own jail | `bootstrap.php` and `wp-cli.yml` are read-only |
| Prompt injection via DB/files | Agent instructions in read-only `CLAUDE.md`; sandbox is disposable |
